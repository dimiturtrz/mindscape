# PyTorch 2.13.0 + CUDA 13.0 (cu130) upgrade audit — for bd h2hq

**Date**: 2026-07-16
**Status**: settled (primary-source facts) + secondary claims flagged
**For**: bd h2hq (torch 2.12.1 → 2.13 upgrade; motivation = unblock the setuptools CVE)

> **Correction note.** The first draft of this file (Haiku researcher) carried two **verified-false**
> load-bearing claims — that no `cp311`/`cp312` cu130 wheels exist, and that torch 2.13 keeps `setuptools<82`.
> Both were checked against primary sources and are wrong (see below). The release-notes / regressions section
> is Haiku-web-sourced and **not** primary-verified — treated as unconfirmed and backstopped by the test gate.

## TL;DR (primary-source verified)

- **Wheels exist for us.** `torch-2.13.0+cu130` win_amd64 wheels are published for **cp310, cp311, cp312,
  cp313, cp314** — we run **Python 3.13.9**, so `cp313` is covered [P1]. Matching `torchvision-0.28.0+cu130`
  cp313 win_amd64 wheel also exists [P2]. The upgrade is **mechanically viable** on the cu130 index.
- **It unblocks the setuptools CVE.** torch **2.12.1** requires `setuptools<82`; torch **2.13.0** requires
  `setuptools>=77.0.3` with **no upper bound** — the `<82` cap is dropped [P3]. So `setuptools>=83`
  (PYSEC-2026-3447 fix) becomes satisfiable after the bump. h2hq's stated motivation **holds**.
- **Remaining gate is behavioral, not availability:** `nox test` + a matched perception re-measure (the geo /
  NICE / foundation numbers) to confirm no numeric drift. That is the real work; it needs GPU.

## Verified facts + sources

### Wheel availability [P1]
`https://download.pytorch.org/whl/cu130/torch/` (fetched 2026-07-16) lists `torch-2.13.0+cu130-cpXXX-...-win_amd64.whl`
for **cp310, cp311, cp312, cp313, cp314** (and cp314t). The earlier "cp311/cp312 missing" claim is false.

### torchvision pairing [P2]
`https://download.pytorch.org/whl/cu130/torchvision/` (fetched 2026-07-16) lists
`torchvision-0.28.0+cu130-cp313-cp313-win_amd64.whl` — the 0.28 ↔ torch-2.13 pairing, cu130, our interpreter.

### setuptools bound [P3]
- `https://pypi.org/pypi/torch/2.13.0/json` → `requires_dist` contains **`setuptools>=77.0.3`** (no `<82`).
- `https://pypi.org/pypi/torch/2.12.1/json` → `requires_dist` contains **`setuptools<82`**.
The cap is removed in 2.13 → the setuptools CVE bump (>=83) is no longer blocked by torch.

## Secondary claims — NOT primary-verified (Haiku web search, treat as unconfirmed)

The first-draft release-notes list (C++20 required to import headers, named-tensors removed, NCCL≥2.23,
bare-PyObject schema restriction, Bazel removed, ROCm CPU `torch.compile` regression, vLLM Inductor/ABI/Dynamo
CI regressions) and the "no torch-2.13-specific incompatibility for mne/braindecode/pyriemann/open-clip/
onnxruntime/onnxscript" claim were **not** checked against the release page directly. The draft also had a
suspicious release date ("2024-07-08") and a wrong "latest stable is 2.7.0" line — discount both.

**Relevance IF the breaking-change list is accurate:** none obviously hits mindscape — we consume **prebuilt
wheels** (C++20 compiler req is N/A), use **no named tensors**, and already **declare onnxscript** as a dep.
Standard ops we rely on (`nn.Conv2d`, `torch.autocast` bf16, `AdamW`, `LambdaLR`, `state_dict`) were not flagged.
Regardless, the `nox test` + matched re-measure gate catches any real breakage empirically.

## Recommendation for h2hq (caller's judgment)

Proceed when a GPU window is free: bump the `[tool.uv] constraint-dependencies` to `torch==2.13.0+cu130` /
`torchvision==0.28.0` (keep `cuda-toolkit` aligned), `uv lock`, then gate on `nox test` + a matched perception
re-measure before merging. Pair it with the setuptools>=83 bump (the CVE this unblocks). Low API-risk, but the
re-measure is mandatory (torch minor bumps can shift kernels / numerics).
